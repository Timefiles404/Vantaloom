# 工程规范

这些规则用于约束 Vantaloom 后续演进，不放在公开 README 中。

## 结构

- 代码按能力边界放入目录，不把源码平铺在单层目录中
- 一个文件只承载一个清晰责任
- 当两个板块可以独立变化时，优先拆成两个文件
- 避免文件膨胀，接近 300 行时主动评估拆分
- 不让应用层、agent 层、协议层和工具层相互穿透

## 产品边界

- Vantaloom 不应被设计成单一桌面应用，而应被设计成 `Vantaloom Core + 多种外壳`
- `apps/desktop` 只是第一个入口，后续可以增加 `apps/server`、`apps/web` 和 `apps/mobile`
- 任何核心能力都不要只写进 `apps/desktop`
- 可复用的 agent 能力优先下沉到 `packages/agent-core`
- 跨桌面、服务端、网页和移动端共享的结构优先下沉到 `packages/protocol`
- UI 外壳只负责交互、展示、输入和平台集成，不直接拥有 agent 核心状态机

## 远程能力

- 远程开发能力应以稳定协议为先，而不是先复制桌面 UI
- 普通请求优先使用 HTTP API，运行事件、日志和任务状态优先使用 WebSocket
- 如果只有服务端到客户端的单向事件流，可以评估 SSE
- 不在早期默认引入 gRPC，除非协议复杂度和多语言代码生成收益已经明确超过维护成本
- 远程执行必须把鉴权、租户隔离、凭据存储、命令权限和审计日志视为核心边界
- GitHub CI 仍然作为远程项目验证和发行的权威环境

## 变更

- 提交信息尽量使用中文
- 每次变更范围保持清晰
- 不做无关重构
- 不提交本地密钥、令牌和机器特定配置

## 验证

- GitHub CI 是正式验证入口
- 本地验证用于快速反馈，不作为发行依据
- 失败日志应沉淀为可解释、可复现的问题
